# X1 录入端保护原则 / 开发红线（2026-05-11）

> 目的：防止再次出现“为了导出顺利而反向污染录入端”的问题，明确 X1 录入端的保护边界、开发红线与正确演进方向。

---

## 一、总原则

### 1. 录入端是事实源头
- 录入端负责承载真实检测事实。
- 判定、导出报告、原始记录、模板填充，都是对事实的消费。
- **任何功能都不得反向改写录入端基础模型。**

### 2. 导出端适配录入端，不是反过来
允许为了导出做：
- 字段映射
- 语义归一
- section 编排
- 模板规则适配
- mixed-report 规则编排

不允许为了导出做：
- 删除录入参数
- 合并录入参数
- 改写录入参数语义
- 改写录入参数布局
- 改变用户录入路径

### 3. 录入端优先级高于导出端
发生冲突时，优先顺序固定为：
1. **保录入端真实性与稳定性**
2. 保判定口径
3. 保导出正确性
4. 保模板适配体验

---

## 二、开发红线

### 红线 1：前端不得用后端 schema 直接覆盖 `params`
**禁止：**
- `params: schema.params`
- 任何等价形式的“后端 schema 决定录入表单结构”

**原因：**
- 这会让录入端表单随规则链、上下文、导出需求而漂移。
- 录入端会失去“稳定事实表单”的属性。

**正确位置：**
- schema 可以存在，但只能服务：
  - 导出编排
  - 判定消费
  - mixed-report 规则消费
  - report context builder

---

### 红线 2：前端不得因标准/判定同步而重建参数区
**禁止：**
- 更新标准范围时重绘整块参数区
- 更新判定优先级时重构表单结构
- 用“前后台同步”为由重建录入主表单

**允许：**
- 更新 range 文本
- 更新标准来源标签
- 更新提示文案

**原则：**
> 范围、标准、判定来源只能影响显示，不能反向影响“录什么”。

---

### 红线 3：前端录入主链不得做参数语义 alias / key 迁移
**禁止：**
- `airchange -> airchange_polluted`
- `pressure -> static_pressure_diff`
- `bacteria -> settling`
- 任何类似“录入语义 → 导出语义”的前端转换

**原因：**
- 这会让用户看到参数像消失、改名、被合并。
- 录入端不再表达原始业务事实，而是表达中间语义层。

**正确位置：**
- 后端标准化层
- 判定层
- 导出层
- report context builder

---

### 红线 4：不得为了模板适配修改录入参数结构
**禁止：**
- 为了某个模板好填而删参数
- 为了某个报告页好拼而并参数
- 为了某个导出对象成功而改布局
- 为了导出通过率而改变录入语义

**正确做法：**
- 改模板规则
- 改导出映射
- 改编排层
- 改 section 生成逻辑
- 改后端消费逻辑

---

### 红线 5：复杂对象不得走“动态拼表单”路线
重点对象：
- `operating_room`
- `animal_room`
- `clean_function_room`
- `negative_pressure`

**禁止：**
- 随上下文切换整套 `params`
- 动态删参 / 补参 / 换 `inputType` 作为录入主链常规做法
- 让房间切换表现得像“规则现场拼表单”

**正确做法：**
- 提供**稳定录入骨架**
- 不适用项可禁用 / 灰显 / 标注说明
- 后端再根据上下文决定哪些字段参与判定和导出

---

## 三、允许做的改动类型

### 1. 纯显示修复
例如：
- 本来应该显示参数区却没显示
- 错误提示“请先选择洁净等级”
- 文案、按钮、展开折叠异常

这类可以直接修。

### 2. 校验增强
例如：
- 必填提示
- 上下文未完成时禁止提交
- 摘要提示链未闭合

这类可以做，但：
- **不能借校验名义去重构参数模型**

### 3. 非语义性布局优化
例如：
- 更清晰的分组
- 行距、间距、折叠优化
- 参数区视觉整理

前提：
- 不改变参数项
- 不改变参数语义
- 不改变录入路径

---

## 四、职责边界

### 录入端负责什么
- 用户录入什么
- 参数怎么展示
- 基础校验
- 结果即时计算展示
- 事实原样提交

### 判定端负责什么
- 标准范围匹配
- 合格/不合格判断
- 标准优先级消费
- 语义标准化

### 导出端负责什么
- section 编排
- 模板字段映射
- 报告拼装
- mixed-report 合并规则
- 文档格式适配

### 原则
> 录入端负责事实，后端负责解释，导出端负责呈现。

## 五、版本回退经验（本次事故结论）

### 本次事故核心问题
录入端曾被引入以下高风险机制：
- 后端 schema 拉取
- `schema.params` 覆盖前端录入参数集
- 标准/判定范围更新触发参数块重绘
- 参数 key alias / 迁移进入录入主链

这些改动导致：
- 参数消失
- 参数增加
- 参数合并
- 参数布局变化
- 用户感觉录入页不再像稳定事实表单

### 本次止血结论
最终确认：
- **最近的干净止血点：`X1_X4.6_manual_backup_20260510_073225.tar.gz`**
- 该版本无 schema 动态改参链
- 回退后录入端已基本恢复正常

### 回退选择原则
以后遇到类似问题：
- 不能只按日期选版本
- 必须按“关键污染链是否存在”选版本

重点检查：
- 是否存在 `fetch...Schema`
- 是否存在 `params: schema.params`
- 是否存在 range update → rerender params
- 是否存在前端主链 key alias / 迁移

## 六、后续 mixed-report 正确路线

mixed-report 不是不能做，但必须按正确方向推进：

### 正确路线
- 录入端保持稳定
- mixed-report 规则放到：
  - 后端规则层
  - 编排层
  - report context builder
  - 导出层

### 错误路线
- 为 mixed-report 先改录入端
- 为导出成功先改参数模型
- 为规则收口先改前端表单

---

## 七、执行要求

以后任何人修改以下文件时，必须先自查是否触碰红线：
- `static/record.js`
- `static/standards_db.js`
- `templates/record_index.html`
- 以及任何影响录入参数结构的前端模块

### 录入端改动授权规则
- 录入端基础模型**默认受保护**。
- 未经刘总明确授意，不得擅自改动：
  - 参数项
  - 参数结构
  - 参数语义
  - 参数布局逻辑
  - 录入路径
- **如刘总明确授意，可以修改录入端**。
- 但每次修改前，**必须先用中文与刘总确认具体改动内容**。
- 确认内容至少包括：
  1. 改动对象
  2. 改动项
  3. 改动方式（删除 / 新增 / 合并 / 改名 / 改布局 / 改路径）
  4. 改动原因
  5. 影响范围
  6. 验证方式
- **未完成中文确认，不得实施。**
- 改动完成后，必须再验证录入端业务正确性，不能只以“导出成功”作为完成标准。

### 改前必答 5 个问题
1. 这次改动是在修显示，还是在改参数模型？
2. 有没有让导出需求反向影响录入结构？
3. 有没有让标准/判定逻辑反向控制录入表单？
4. 有没有把 alias / schema / 编排逻辑放进前端主链？
5. 如果删掉导出需求，这个改动还有必要存在吗？

只要第 2、3、4 任意一项回答为“有”，就必须停下来重审。

---

## 八、一句话原则

> **X1 的真正生命线不是报告导出，而是录入端的真实性与稳定性。**

报告错了，可以重导。
模板错了，可以重调。
但如果录入端被带歪，整个系统都会失去意义。
