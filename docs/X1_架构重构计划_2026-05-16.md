# X1 系统架构重构计划

**制定日期**: 2026-05-16  
**当前状态**: 功能完整度 83/100，架构健康度 42/100  
**目标**: 在不中断业务的前提下，分阶段将架构健康度提升到 70+

---

## 重构原则

1. **渐进式**：每个阶段独立可交付，不做大爆炸重写
2. **功能不退化**：每阶段完成后跑回归测试，确保零功能损失
3. **先拆后治**：先解决文件级耦合，再解决逻辑层耦合
4. **业务优先**：如果有紧急业务需求，重构可暂停，但不回退

## 阶段一：后端 Blueprint 拆分（预计 2-3 天）

### 目标
将 app_x1.py 从 7020 行 / 125 路由 拆分为 8 个 Blueprint 模块 + 1 个精简主文件。

### 拆分方案

```
app_x1.py (精简后 ~1500 行)
  ├── Flask 初始化、中间件、CSRF、全局配置
  ├── 首页/静态路由
  ├── 认证路由 (login/register/logout)
  └── Blueprint 注册

routes/
  ├── __init__.py
  ├── template_mgmt.py    (~1800行, 21路由) — 模板管理全部
  ├── settings.py         (~600行, 13路由)  — 系统设置+备份还原
  ├── projects.py         (~700行, 12路由)  — 项目管理CRUD
  ├── tasks.py            (~500行, 10路由)  — 任务派单+执行
  ├── records.py          (~800行, 8路由)   — 记录管理+批量操作
  ├── export.py           (~600行, 5路由)   — 导出/报告生成
  ├── drafts.py           (~400行, 7路由)   — 草稿管理
  └── admin_misc.py       (~500行)          — 用户/权限/日志/标准/统计/文件

helpers/
  ├── __init__.py
  ├── db.py               — 统一数据库访问（get_db + get_x1_data_conn）
  ├── project_utils.py    — 项目/任务相关辅助函数
  ├── template_utils.py   — 模板相关辅助函数
  └── export_utils.py     — 导出相关辅助函数
```

### 执行步骤

1. 创建 `routes/` 和 `helpers/` 目录
2. 提取共享辅助函数到 `helpers/`（74 个非路由函数）
3. 按模块逐个提取路由到 Blueprint（从最独立的开始）
4. 提取顺序：`settings` → `projects` → `tasks` → `records` → `drafts` → `export` → `template_mgmt` → `admin_misc`
5. 每提取一个模块后重启验证
6. 最后精简 app_x1.py 为注册入口

### 验证标准
- 服务启动正常
- 健康检查 PASS
- 所有路由可达（curl 测试）
- 回归测试通过

## 阶段二：前端模块化（预计 2-3 天）

### 目标
将 record.js (7806行) 拆分为可维护的模块，引入轻量构建。

### 拆分方案

```
static/src/
  ├── record/
  │   ├── index.js          — 入口，初始化
  │   ├── domain.js         — 领域/对象选择
  │   ├── room.js           — 房间管理（添加/删除/等级选择）
  │   ├── params.js         — 参数渲染（20种 inputType）
  │   ├── collect.js        — collectData()
  │   ├── restore.js        — restoreDraft()
  │   ├── calc.js           — 计算函数（换气次数/体积/修正等）
  │   ├── export.js         — 导出提交逻辑
  │   └── ui.js             — 通用 UI 辅助
  ├── admin/
  │   ├── index.js          — 后台入口
  │   ├── projects.js       — (现有 admin_projects.js)
  │   ├── customers.js      — (现有 admin_customers.js)
  │   ├── stats.js          — (现有 admin_stats.js)
  │   └── templates.js      — 模板管理前端逻辑（从 admin.html 提取）
  └── customer/
      └── index.js          — (现有 customer.js)

static/dist/
  ├── record.bundle.js      — esbuild 打包输出
  ├── admin.bundle.js
  └── customer.bundle.js
```

### 构建工具
- **esbuild**：零配置、极快（<100ms 打包）、无 node_modules 依赖地狱
- `package.json` 加 `"build": "esbuild static/src/record/index.js --bundle --outfile=static/dist/record.bundle.js"`

### 执行步骤
1. 安装 esbuild
2. 从 record.js 中提取 `collectData` → `collect.js`（最独立的函数）
3. 逐步提取其他模块
4. HTML 模板引用改为 `dist/record.bundle.js`
5. 保留原 record.js 作为 fallback 直到验证完成

## 阶段三：数据层统一（预计 1-2 天）

### 目标
消除双数据库散落调用，建立统一数据访问层。

### 方案

```python
# helpers/db.py

class X1Database:
    """统一数据访问层"""
    
    @contextmanager
    def main_db(self):
        """主库连接（用户/日志/通知/记录）"""
        ...
    
    @contextmanager  
    def business_db(self):
        """业务库连接（项目/客户/任务/反馈）"""
        ...
    
    # 项目相关
    def get_project(self, project_id): ...
    def list_projects(self, filters): ...
    def create_project(self, data): ...
    
    # 任务相关
    def get_task(self, task_id): ...
    def create_task(self, data): ...
    
    # 记录相关
    def get_record(self, record_id): ...
    def list_records(self, filters): ...
```

### 执行步骤
1. 创建 `helpers/db.py` 统一封装
2. 逐步将路由中的直接 SQL 调用替换为 db 方法
3. 明确文档化哪些表在哪个库
4. 长期目标：合并为单库（SQLite 无跨库事务）
