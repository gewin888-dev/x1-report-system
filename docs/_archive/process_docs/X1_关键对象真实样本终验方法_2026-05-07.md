# X1 关键对象真实样本终验方法（2026-05-07 迭代版）

## 核心原则

不是先看页面绿不绿，而是按四层事实往下打：

1. **语义层**：`semantic key` 是否命中正确业务场景
2. **模板层**：`template key` 是否命中正确模板链
3. **资源层**：模板文件是否真实存在、资源注册是否闭合
4. **产物层**：`.filled.docx` 是否真生成、基础字段是否真写入

---

## 标准打法（固定顺序）

### 第一步：哨兵验证
先挑最能代表主链的少量样本，验证：
- `semantic key`
- `template key`
- `resource confirmed`
- `template found`

目标：
- 先确认主链是否闭合
- 不在主链没通时提前陷进内容细节

### 第二步：扩展场景验证
如果对象有多个业务分支 / 多场景：
- 把剩余关键场景批量补齐
- 优先检查语义链与模板链是否全覆盖

目标：
- 防止“哨兵通过，但某些子场景仍掉旧链”

### 第三步：内容探针
在主链通过后，再实际生成 `.filled.docx`，检查：
- 项目名称
- 报告编号
- 客户名
- 房间名
- 温度 / 湿度 / 噪声 / 照度等基础字段

目标：
- 把“命中了模板”推进到“模板里真的有内容”

---

## 当前已验证出的高频问题类型

### 1. semantic mapping 默认 key 被历史别名污染
表现：
- 规则本体已正确
- 但 `default_template_key` 还指向旧短 key / 过程 key

处理：
- 先修 `template_semantic_mappings.json`
- 必要时同步修 `template_type_mappings.json`

### 2. template resource 文件名还是旧口径
表现：
- `semantic key` 对
- `template key` 对
- 但 `template_found` 为 false

处理：
- 直接核机器上真实文件名
- 修 `template_resources.py`
- 不凭旧记忆猜文件名

### 3. 多场景对象看似多模板，实际是共享模板
表现：
- 子场景很多
- 资源定义误以为每个子场景都有独立物理模板

处理：
- 先查真实目录
- 接受“多场景 → 同一物理模板”的真实结构
- 不强行做一场景一物理模板假设

### 4. 前台参数多入口 ≠ 模板端多落点
表现：
- 录入端参数很多
- 容易误判模板必须有多个对应坑位

处理：
- 以业务规则为准
- 典型例子：医院洁净部照度 = **多入口单落点**

---

## 已沉淀出的质量提升策略

### 策略 A：先打主链，再打内容
不跳步。主链没闭合前，不把精力耗在内容细节上。

### 策略 B：每修一层就立刻复跑
不要堆一堆修改后再一起验证，否则定位会变脏。

### 策略 C：优先接受真实资源结构
真实目录里如果就是共享模板，就按共享模板落档，不做臆想性“应有独立模板”。

### 策略 D：把业务解释写进系统
发现口径后，不只口头记住，要：
- 写代码注释
- 写规则文档
- 写终验结论

### 策略 E：把“通过”拆层表达
- 页面治理通过
- 语义命中通过
- 模板命中通过
- 资源存在通过
- 内容抽样通过

不能混成一个“已完成”。
