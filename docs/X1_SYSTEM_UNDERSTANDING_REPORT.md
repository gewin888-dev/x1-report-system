# X1 系统认知报告（正式版）

更新时间：2026-05-20

---

## 一、文档目的

本报告用于从全局视角说明 X1 检测报告生成系统的真实结构、业务组成、关键链路、数据分层和高风险区域，作为后续系统巡检、修复、治理和版本决策的基础文档。

本报告不是某个局部功能说明，而是面向系统级理解，目标是回答以下问题：

1. X1 系统实际上由哪些部分组成
2. 员工端、客户端、后台管理端如何协同
3. 核心业务主链是什么
4. 数据、文件、模板、飞书、项目状态之间如何联动
5. 当前系统最容易出问题的地方在哪里

---

## 二、系统定位

X1 不是单纯的“表单导出工具”，而是一个围绕洁净检测业务全流程运作的综合系统。

从代码与运行结构看，X1 至少覆盖以下业务能力：

- 检测部员工录入检测项目和房间参数
- 保存草稿与恢复录入状态
- 生成检测报告（Word）
- 生成原始记录（Excel）
- 模板命中与模板治理
- 项目管理与任务派单
- 客户门户查看项目与历史记录
- 客户催单、反馈、确认
- 飞书上传交付
- 正式归档
- 后台治理、监控与备份恢复

因此，X1 更接近于：

> 洁净检测业务中台 + 报告生成引擎 + 客户交付门户 + 后台治理系统

---

## 三、系统三端结构

### 1. 员工端 / 使用端

员工端是检测部日常使用的主入口，核心页面和接口由 `app_x1.py` 直挂入口，并由 `drafts`、`export` 等蓝图提供能力。

主要入口：
- `/login`
- `/`
- `/home`
- `/api/save`
- `/api/list`
- `/download/<filename>`
- `/api/preview/<filename>`

主要页面：
- `login.html`
- `landing.html`
- `record_index.html`

员工端核心职责：
- 登录系统
- 录入检测数据
- 保存草稿
- 恢复草稿
- 导出检测报告
- 导出原始记录
- 下载与预览文件

---

### 2. 客户端 / 客户门户

客户端是独立门户，不是后台中的嵌套模块。

主要入口：
- `/customer/login`
- `/api/customer_login`
- `/customer`

主要页面：
- `customer_login.html`
- `customer.html`

客户端核心职责：
- 客户登录
- 查看项目进度
- 查看历史记录
- 下载或查看报告结果
- 发起催单
- 提交反馈
- 确认报告
- 维护客户资料

---

### 3. 后台管理端

后台管理端由一个主页面承载，但功能本体分散在多个业务蓝图中。

主要入口：
- `/admin`

主要页面：
- `admin.html`
- `settings.html`
- `templates.html`
- `standards.html`
- `monitor.html`

后台核心职责：
- 报告管理
- 项目管理
- 用户管理
- 模板管理
- 标准数据库管理
- 系统设置
- 系统监控与日志
- 备份恢复
- 客户与任务协同管理

## 四、后端结构与蓝图分工

### 1. 主装配入口：`app_x1.py`

`app_x1.py` 仍是系统总装配中心，负责：
- Flask 应用初始化
- 配置读取
- 目录初始化
- 部分主入口路由
- 蓝图注册
- 仍残留少量业务逻辑与管理接口

当前它的特点是：
- 已开始模块化拆分
- 但仍承担较多主入口责任
- 是后续结构治理的重要对象之一

---

### 2. 路由蓝图划分

#### `routes/drafts.py`
负责草稿链：
- 保存草稿
- 读取草稿
- 列出草稿
- 草稿转正式记录

#### `routes/export.py`
负责导出主链：
- 原始记录导出
- 检测报告导出
- 构建 export payload
- 旧接口兼容
- 导出后异步后处理

#### `routes/records.py`
负责后台记录管理：
- 记录列表
- 汇总统计
- 飞书重试
- 作废导出
- 批量删除
- 回收站状态

#### `routes/projects.py`
负责项目管理：
- 项目 CRUD
- 项目汇总
- 项目关联报告
- 文件下载
- 历史报告上传
- 检测员/客户列表

#### `routes/tasks.py`
负责任务管理与检测员任务链：
- 项目任务 CRUD
- 检测员待办任务
- 接单/开始/完成
- 预填项目基础信息

#### `routes/settings.py`
负责系统设置与运维：
- 系统设置
- 路径巡检
- 飞书测试
- 备份与恢复
- 健康检查

#### `routes/template_mgmt.py`
负责模板治理：
- 模板主页
- 模板注册
- 模板上传
- 模板验证
- 默认模板
- 语义模板
- 模板变量查看

#### `routes/admin_misc.py`
负责后台杂项治理：
- 通知
- 注册审核
- 用户管理
- 权限配置
- 日志
- 标准库
- 统计

#### `customer_routes.py`
负责客户端核心逻辑：
- 客户门户页面
- 客户资料
- 客户项目
- 历史记录
- 催单
- 反馈
- 报告确认

## 六、数据分层结构

X1 当前采用双数据库分层模型。

---

### 1. 主库：`x1_data.db`

负责系统治理与记录元数据。

主要表：
- `users`
- `role_permissions`
- `system_settings`
- `action_logs`
- `error_logs`
- `performance_logs`
- `inspection_records`
- `notifications`
- `customer_registrations`

主要职责：
- 用户认证与角色权限
- 系统设置
- 运行日志
- 通知
- 客户注册审核
- 检测记录元数据

---

### 2. 业务库：`data/x1_data.db`

负责项目、任务与客户协同数据。

主要表：
- `business_projects`
- `project_tasks`
- `client_profiles`
- `project_urge_logs`
- `client_feedback`
- `report_feedback`

主要职责：
- 项目管理
- 任务执行流
- 客户资料
- 客户催单
- 客户反馈
- 报告确认反馈

---

### 3. 双库结构的业务含义

X1 不是单库应用，真实状态分散在：
- 主库
- 业务库
- 草稿 JSON
- 导出 JSON
- Word / Excel 文件
- 飞书链接
- 正式归档目录

因此系统巡检和修复不能只看数据库，必须同时看：
- 数据库
- 文件系统
- 页面
- 接口
- 外部上传结果

---

## 七、核心业务对象

### 1. 项目对象 `business_projects`
项目是整个业务协同层的主对象，承载：
- 项目信息
- 客户信息
- 检测领域与对象类型
- 检测执行状态
- 报告状态
- 财务状态
- 催单状态
- 文件关联状态

它不是简单项目表，而是：
> 经营状态 + 检测状态 + 报告交付状态 的综合业务对象

---

### 2. 任务对象 `project_tasks`
任务对象用于管理检测员执行流，具备：
- 接单
- 开始执行
- 完成
- 取消
- 备注
- 时间戳

任务状态机至少包括：
- `pending_assign`
- `assigned`
- `accepted`
- `in_progress`
- `completed`
- `cancelled`

说明 X1 已具有流程化派单能力。

---

### 3. 记录对象 `inspection_records`
主库存储的是记录元数据，但后台记录页真正使用的是“聚合记录资产对象”，而不是单表直读。

一条记录可能包含：
- 草稿 JSON
- 标准导出 JSON
- Word 报告
- Excel 原始记录
- 飞书报告状态
- 飞书原始记录状态
- 作废状态

---

### 4. 房间对象 `room`
导出链中真正的判定与模板命中最小单位是房间，而不是项目。

每个项目可包含多个房间，每个房间独立经历：
- 参数校验与补全
- 判定引擎计算
- 模板命中
- 模板资源解析
- 报告上下文构建

---

### 5. 客户对象
客户对象分散在两层：
- 主库 `users.client_name`
- 业务库 `client_profiles`

并通过客户名称与项目、反馈、催单、确认记录关联。

---

## 八、六条核心业务链

### 链 1：员工录入与导出链
员工登录 → 录入项目/房间参数 → 保存草稿 → 标准化 payload → 房间级判定 → 模板命中 → 生成报告/原始记录

这是检测部日常使用的第一核心链。

---

### 链 2：导出后交付链
导出成功 → 飞书上传报告与原始记录 → 正式归档 → PDF 预览 → sidecar JSON 回写

这条链是导出成功后的异步交付链。

---

### 链 3：记录管理聚合链
后台记录页聚合：
- 草稿
- 导出 JSON
- Word/Excel 文件
- 飞书状态
- 资产健康状态

它本质上是一个交付健康聚合层。

---

### 链 4：项目与任务联动链
任务动作和导出结果都会反向推进项目状态：
- 接单 → 已排期
- 开始执行 → 检测中
- 任务完成 → 检测完成 + 编制中
- 导出成功 → 检测完成 + 报告编制中

说明项目状态至少有双推进源。

---

### 链 5：客户门户链
客户登录 → 查看项目 → 查看历史记录 → 催单 → 反馈 → 报告确认

客户端不是只读结果页，而是轻量协同门户。

---

### 链 6：备份恢复兜底链
- code 备份/恢复
- data 备份/恢复
- full 备份/恢复
- 恢复前快照
- 重启与健康检查

这是系统级容灾兜底链。

---

## 九、当前最值得警惕的系统级风险

### 风险 1：双数据库口径问题
主库与业务库边界明确，但任何迁移、备份、恢复、查询若漏一边，就会造成口径不一致。

### 风险 2：导出链耦合过深
导出链同时耦合：
- 数据标准化
- 房间级判定
- 模板规则
- 文档生成
- 飞书上传
- 归档
- 项目状态推进

这是全系统最核心也最危险的链路。

### 风险 3：记录页不是单数据源
后台记录管理页很多问题不是数据库问题，而是草稿、导出结果、文件资产、飞书状态、模板状态的聚合问题。

### 风险 4：项目状态有双推进源
任务链推进与导出链推进都能改变项目状态，容易出现竞争、漏同步、状态冲突。

### 风险 5：客户端是双源聚合视图
客户历史与客户项目并不完全来自同一数据源，容易出现重复、遗漏或状态分类错误。

### 风险 6：文件系统是真实业务资产层
很多“真正交付结果”在文件系统与飞书链接，而不在数据库本体中。

---

## 十、结论

X1 已经不是简单 Flask CRUD 项目，而是一个多端、多链路、双数据库、文件资产驱动、带模板治理和客户协同能力的综合业务系统。

因此后续任何巡检、修复、版本治理，都必须以“系统级视角”推进，而不是局部页面修补。

后续正式巡检与修复计划，必须建立在以下前提上：
1. 同时检查员工端、客户端、后台端
2. 同时检查数据库、文件系统、页面、接口、飞书状态
3. 优先保护导出主链与项目状态推进链
4. 所有修复都要考虑双数据库与文件资产的影响面
