## 二、安全审计

### 2.1 高危问题

**无高危问题。**

系统整体安全基线合格：
- 密码使用 `pbkdf2:sha256` 哈希存储 ✅
- CSRF 防护已实现（Origin/Referer 校验） ✅
- 登录限流已实现（IP 级别） ✅
- Debug 模式已关闭 ✅
- 敏感配置文件已加入 .gitignore ✅

### 2.2 中危问题

| # | 文件:行 | 问题 | 风险 |
|---|---------|------|------|
| M1 | `app_x1.py:1539-1545` | workspace_doc API 接受任意绝对路径，仅靠文件名白名单防护。攻击者可构造 `path=/etc/passwd` 但文件名为白名单中的名字（如符号链接场景） | 路径遍历（需认证+白名单，实际利用难度高） |
| M2 | `static/admin.js:1060-1090` | `renderRecords()` 中 `r.project_name`、`r.client_name`、`r.operator` 等字段直接拼入 innerHTML，未调用 `escapeHtml()` | 存储型 XSS（需要先注入恶意数据到记录中） |
| M3 | `helpers/db.py:175,184,191,199,204` | `x1_count/x1_get_by_id/x1_insert/x1_update/x1_delete` 使用 f-string 拼接表名。虽然调用方传入的是硬编码表名，但函数本身不做校验 | SQL 注入（间接风险，当前调用安全） |
| M4 | `routes/settings.py:43` | 路径浏览白名单包含 `Path.home()`，允许浏览整个用户目录 | 信息泄露（需 admin 权限） |

### 2.3 低危问题

| # | 文件:行 | 问题 | 风险 |
|---|---------|------|------|
| L1 | `app_x1.py:659` | `secret_key` 回退到自动生成文件，但 `x1_config.json` 未在 .gitignore 中 | 如果 config 中写入 secret_key 可能泄露 |
| L2 | `app_x1.py:666` | `SESSION_COOKIE_SECURE=False`（HTTP 环境合理，但局域网外访问时 cookie 可被截获） | Session 劫持（仅非 HTTPS 环境） |
| L3 | `app_x1.py:700` | CSRF 校验对 `is_json` 请求直接放行（`return bool(request.is_json)`），攻击者可伪造 Content-Type | CSRF 绕过（需跨域 + JSON body） |
| L4 | 多处 | `except Exception: pass` 静默吞没异常（13处），可能掩盖运行时错误 | 调试困难 |

## 三、代码质量

### 3.1 架构评估

**优点：**
- 路由模块化拆分合理（routes/ 目录按功能分 9 个 Blueprint）
- 数据访问层有统一封装（helpers/db.py）
- 操作日志覆盖全面（登录/保存/导出/删除/模板/用户/设置）
- 权限系统完整（角色 + 细粒度权限 key）
- 配置外部化（x1_config.json）

**不足：**
- `app_x1.py` 仍有 1,582 行，承担了过多职责（路由 + 工具函数 + 配置加载 + 业务逻辑）
- 前端为单文件巨型 JS（admin.js 3,316 行），无模块化
- `templates/admin.html` 4,270 行，CSS/HTML/JS 混合
- 双数据库设计（x1_data.db + data/x1_data.db）增加了维护复杂度

**架构健康度**: 58/100
