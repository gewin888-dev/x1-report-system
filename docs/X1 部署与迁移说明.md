更新时间：2026-05-22

---

## 一、适用范围

本说明用于：
- 把 X1 迁移到新机器
- 恢复一套可运行的本地单机环境
- 核对项目目录、外部模板目录、备份目录、启动脚本是否完整

## 三、需要一起迁移的内容

### 1. 项目目录
至少包括：
- 代码
- `x1_data.db`
- `data/x1_data.db`
- `records_x1/`
- `reports_x1/`
- `templates/`
- `static/`
- `logs_x1/`（可选保留）
- `docs/`

### 2. 配置与凭证
重点检查：
- `x1_config.json`
- 模板注册相关配置
- 外部目录路径配置

### 3. 外部业务目录
如仍依赖本机目录，还要一起迁移：
- 检测报告模板目录
- 正式报告归档目录
- 原始记录归档目录
- 备份目录

## 四、迁移步骤建议

### Step 1：复制项目目录
复制当前 X1 项目目录到目标机器。

### Step 2：复制双数据库与运行数据
必须确认以下数据一并带走：
- 根库 `x1_data.db`
- 业务库 `data/x1_data.db`
- 草稿目录 `records_x1/`
- 导出目录 `reports_x1/`

### Step 3：复制模板与归档目录
如果模板和归档仍挂在外部目录，必须同步迁移。

### Step 4：更新配置路径
重点核对：
- 模板根目录
- 正式报告归档目录
- 原始记录归档目录
- 备份目录

### Step 5：启动并验活
使用标准方式：
```bash
bash start_x1_daemon.sh
curl http://127.0.0.1:8082/api/x/health
bash doctor_x1_daemon.sh
```

### Step 6：最小业务验证
至少验证：
- 登录
- 后台可打开
- 员工端可打开
- 项目列表可读
- 任务列表可读
- 模板可见
- 健康检查正常

## 五、迁移后重点注意

### 1. 双库不要漏
X1 不是单库系统，迁移时如果只带走一个库，系统会表现为“部分功能正常、部分数据丢失”。

### 2. 前端运行真入口不要误判
当前员工端真实运行脚本是：
- `static/record.js`

不要把仅存在的构建产物当成迁移后唯一验证依据。

### 3. 权限真相不要看旧表
迁移后如果核对权限状态，应优先看：
- `role_permission_final`

不是旧的 `role_permissions`。

### 4. 重启方式统一
迁移后所有标准启停、诊断、恢复都应尽量通过项目自带脚本完成，不要长期依赖手工 python 前台起服务。
